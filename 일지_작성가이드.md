# 업무일지 작성 가이드 (코덱스용)

> 목적: 실험 반복마다 **결과 + 그 원인 + 인사이트**를 사람이 쓴 것처럼 `업무일지.md`에 이어 쓰기.
> 핵심은 "표를 붙이는 것"이 아니라 **왜 그런 결과가 나왔는지 설명**하는 것.

## 0. 대원칙
- **1인칭, 솔직하게.** "잘 됐다"보다 "왜 됐/안 됐는지"를 쓴다. 실수·삽질·예상 빗나감을 숨기지 말 것 — 그게 평가에서 가장 값진 부분.
- **결과 → 원인 → 결정** 흐름. 지표 숫자를 던지지 말고, 그 숫자가 **왜 움직였는지** 데이터로 설명한다.
- **날짜는 유연하게.** 꼭 하루 단위/연속일 필요 없음. 실험 사이클마다 또는 작업 세션마다 한 항목. 각 항목에 날짜만 달면 됨(예: `2026-07-08 오후`).
- 완벽한 정리보다 기록의 밀도. 그날 생각·감정을 녹인다.

## 1. 원인은 '질문별 상세'에서 캐낸다
집계 지표(coverage 0.5 같은)만 보면 원인을 모른다. 반드시 아래를 열어 **어느 질문이 왜 그랬는지** 확인:
- `eval/results/experiments_*_detail.json` — 실험별·질문별 `coverage / first_hit_rank / abstention / retrieved_orgs / answer / judge`.
- run_experiments 콘솔의 `[qN] cov=.. rank=.. abstain=..` 줄.

**원인 분석 예시 논리:**
- "전체 coverage가 0.5인데, 상세를 보니 q2·q5(후속질문)만 0이고 단일질문은 1.0. → 후속질문에 기관명이 없어 검색이 샌 것. 쿼리 재작성이 필요."
- "MMR을 켜니 q1이 갑자기 abstain=True. → 다양성 강제가 단일문서 질문에서 최적 청크를 밀어내 답변 근거가 약해진 것."
- "자동필터로 q4가 cov 0→1, rank 1. → 발주기관 메타 필터가 정답 문서를 최상위로 끌어올림."
→ 이렇게 **집계 변화 ↔ 특정 질문의 거동 ↔ 메커니즘**을 연결해서 쓴다.

## 2. 한 항목의 구성 — **코드잇 공식 양식(6항목) 준수**
가이드의 "Daily 업무일지 양식" 필수 6항목을 따른다. 원하던 "결과+원인 분석"은 [2]와 [4]에 녹인다.
```
## YYYY-MM-DD · <이번 사이클 주제>

**[1] 오늘 날짜**: YYYY-MM-DD

**[2] 오늘 수행한 주요 업무**
- 바꾼 설정/파라미터, (재빌드 시 청크수·비용), 실행한 실험.
- 결과(핵심 지표): coverage __ / mrr __ / groundedness __ / relevance __ / latency __
- 전 사이클 대비 변화.

**[3] 오늘 작업 완료도**: __% — 근거: <간단히>
  (0 / 25 / 50 / 75 / 100 중 하나)

**[4] 오늘 업무 중 발견 / 인사이트**  ← 원인 분석을 여기에
- 질문별 상세를 근거로 **왜 이 숫자가 나왔는지**. 어느 질문이 좋아지고 나빠졌는지 + 메커니즘.
- 그로부터 내린 의사결정(채택/폐기 + 이유).

**[5] 오늘 어려웠던 점**
- 막힌 점, 예상 빗나감, 버그와 해결.

**[6] 내일 목표 / 할 일**
- 다음에 시도할 실험/작업.
```

## 3. 좋은 예시 (이 밀도를 목표로)
```
## 2026-07-08 · 원본(raw) 재빌드 후 baseline 재측정

**[1] 오늘 날짜**: 2026-07-08

**[2] 오늘 수행한 주요 업무**
- build_index raw 재실행. 청크 616 → 11,842개(문서당 118개), 임베딩 약 240만 토큰.
- 검색 설정은 이전과 동일(7개 세트)하게 두고 데이터 소스만 raw로 바꿔 '통제 비교'.
- 결과: groundedness 2.0 → 3.9, relevance 1.9 → 3.7 상승. coverage 0.62 → 0.55 소폭 하락.

**[3] 오늘 작업 완료도**: 75% — 근거: 원본 baseline 측정·비교는 끝, 최종 조합 확정과 보고서는 남음.

**[4] 오늘 업무 중 발견 / 인사이트**
- 상세(detail.json)를 보니 품질 상승은 원문에 세부 요구사항이 들어와 모델이 근거를 인용할 수 있게 된 덕분.
  coverage가 내린 건 한 문서가 118청크로 쪼개져 top_k가 '같은 문서'에 몰려, 비교질문(q7/q8)에서 두 기관을
  동시에 담기 어려워졌기 때문. 즉 원본은 '깊이'를 얻고 '다양성'을 잃는 트레이드오프.
- 결정: 원본 채택(품질 이득이 큼). 비교질문 대응으로 자동필터($in)는 필수 유지.

**[5] 오늘 어려웠던 점**
- 처음엔 coverage가 떨어져 "원본이 나쁜가?" 했는데, 질문별로 뜯어보니 품질은 크게 오른 거였다.
  집계 지표 하나로 판단하면 안 된다는 걸 또 배움.

**[6] 내일 목표 / 할 일**
- top_k를 키워 다양성 손실을 보완할 수 있는지 스윕. 최종 조합 확정 후 보고서 결과 파트 작성.
```

## 4. 하지 말 것
- 표만 붙이고 원인 없이 넘어가기. (표는 experiment_log.md가 이미 함)
- 커버리지 0인데 groundedness 높다고 "성공"이라 쓰기 → 빈 인덱스 착시(CLAUDE.md §7-6). coverage 0이면 실패로 간주.
- 이전 항목 덮어쓰기. 항상 **append**.

## 5. 평가 상승 기록 원칙
- 목표는 "좋은 점수 하나"가 아니라 **평가지표가 어떻게 올라갔는지의 경로**를 남기는 것이다.
- 점수가 오른 실험은 `무엇을 바꿨는지 → 어떤 지표가 얼마나 올랐는지 → 왜 오른 것 같은지 → 채택 여부`를 쓴다.
- 점수가 떨어진 실험도 반드시 쓴다. `어떤 지표가 떨어졌는지 → 어느 질문에서 깨졌는지 → 원인이 검색/생성/평가/데이터 중 무엇인지 → 다음 실험에서 어떻게 보완할지`를 남긴다.
- 실험마다 EDD score, coverage, MRR, groundedness, relevance, abstention, latency를 같이 본다. 단일 지표만 보고 성공/실패를 판단하지 않는다.
- 가능한 시각화는 모두 남긴다. 최소 EDD 막대그래프, metric heatmap, retrieval-quality 산점도, latency-vs-score 그래프를 생성하고 일지에는 그래프에서 읽은 인사이트를 문장으로 적는다.
- 일지는 기술 보고서처럼 딱딱하게 쓰지 말고, "처음엔 이렇게 생각했는데 숫자를 보니 아니었다"처럼 실제 판단 변화를 일기처럼 남긴다.
- 최종 후보는 가능하면 5.5급 LLM 감리로 실제 문맥 품질을 다시 본다. EDD가 높아도 답변이 질문 의도와 어긋나면 그 어긋남을 일지에 쓴다.
- 보고서에 쓸 수 있게 질문별 답변 샘플도 같이 남긴다. 특히 "점수는 높지만 납득 안 됐던 답변", "점수는 낮지만 실제로는 정상 부정 답변이었던 사례", "그 사례 때문에 다음 실험을 바꾼 이유"를 일기처럼 적는다.
- 질문도 가능하면 강한 추론 모델로 다양하게 만든다. 단, 답변은 평가 대상 RAG가 하게 두고, 품질 감리는 다시 강한 모델이 한다. 출제자/응답자/채점자를 섞지 않는다.
