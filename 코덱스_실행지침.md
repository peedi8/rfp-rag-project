# 코덱스 실행 지침 (실험 → 기록 → 일지 반복 루프)

> 이 파일은 **로컬 실행 에이전트(코덱스)** 전용 작업 지시서다.
> 사용자는 이 루프를 코덱스에 맡기고, 보고서 등 다른 작업은 Cowork(다른 에이전트)와 진행한다.
> **시작 전 반드시 읽을 것: (1) `CLAUDE.md`(전체 맥락) (2) `일지_작성가이드.md`(일지 쓰는 법).**

## 역할 경계 (충돌 방지)
- **코덱스(너)가 담당**: 실험 실행, `eval/experiment_log.md`(자동 로그), `업무일지.md`(서술 일지) 갱신.
- **Cowork가 담당**: 보고서 PDF, 코드 구조 리뷰. → `보고서*.md/pdf`는 건드리지 말 것.
- 파일 편집은 **append 위주**로 하고, `src/` 로직을 바꿀 땐 CLAUDE.md에 근거를 남길 것.

## 매 실험 사이클 (이걸 반복한다)
1. **다음 실험 하나 선택** (아래 §백로그에서 위→아래 순서로).
2. 필요한 설정 변경:
   - 검색시점 옵션(top_k, mmr, 필터, 리랭크, 쿼리재작성)만 바꾸면 → `scripts/run_experiments.py`의 `EXPERIMENTS` dict 수정. **재빌드 불필요.**
   - 청크크기(`CHUNK_SIZE`)·임베딩모델·데이터소스(csv/raw)를 바꾸면 → `config.py` 수정 후 **`python -m scripts.build_index [raw|csv]` 재실행 필요.**
3. 실행: `python -m scripts.run_experiments`
   - ✅ 시작 시 "인덱스 청크 수: N" (0이면 즉시 중단하고 build_index부터).
4. 결과 누적: `python -m scripts.log_run`  → `eval/experiment_log.md`에 표+자동관찰 append.
5. **업무일지 갱신 (가장 중요)**: `일지_작성가이드.md`를 따라 `업무일지.md`에 항목을 append.
   - 표만 붙이지 말 것. **결과 + 왜 그런지(원인) + 결정**을 서술한다.
   - 원인은 `eval/results/experiments_*_detail.json`(질문별 상세)을 열어 근거로 삼는다.
   - 1인칭·솔직한 톤, 날짜만 달면 됨(연속일·7일 형식 불필요).
6. 결과를 사용자에게 1~2줄로 요약 보고. 다음 사이클로.

### 가드레일 (반드시 체크)
- build 후 "저장 완료: N개 청크" 문구 확인. raw면 N=11842 근처.
- 실험표에서 **모든 coverage가 0이면 = 빈 인덱스 착시**. 절대 "성공"으로 기록하지 말고 build_index부터 다시.
- groundedness/relevance가 None이면 judge 토큰/JSON 문제 → CLAUDE.md §7-2 참조.
- 원본 재빌드는 실제 임베딩 비용(~$0.05, 240만 토큰) 발생 — 청크/임베딩/소스 안 바뀌면 재빌드 금지(불필요).

## 실험 백로그 (위에서부터 순서대로)
1. **raw 베이스라인 확정** — 현재 7개 설정을 raw 인덱스에서 실행. CSV 대비 groundedness/relevance 상승 확인. 실험7(쿼리재작성)에서 후속질문 coverage 회복 확인.
2. **top_k 스윕** — 5 / 8 / 12 비교. 원본은 한 문서가 100+청크라 top_k가 한 문서에 몰릴 수 있음 → 비교질문 영향 관찰.
3. **mmr_lambda 스윕** — 0.3 / 0.5 / 0.7 (관련성 vs 다양성).
4. **chunk_size 스윕** — 500 / 800 / 1200 (각각 build_index raw 재실행 필요). 세부 요구사항 포착력 vs 노이즈 트레이드오프.
5. **fetch_k 스윕** — 20 / 40 (MMR 후보 풀 크기).
6. **자동필터 on/off × 쿼리재작성 on/off** 2x2 — 최종 조합 확정.
7. **최종 설정 픽스** — 가장 좋은 조합을 `config.py` 기본값 + `RAGPipeline` 기본 params로 반영. 근거를 CLAUDE.md §6에 추가.

## 업무일지 양식
**코드잇 공식 6항목 양식**을 따른다. 상세·예시는 `일지_작성가이드.md` 참조.
```
## YYYY-MM-DD · <주제>
**[1] 오늘 날짜**: YYYY-MM-DD
**[2] 오늘 수행한 주요 업무**  — 설정/실험 + 결과 지표 + 전 대비 변화
**[3] 오늘 작업 완료도**: __% — 근거
**[4] 오늘 업무 중 발견/인사이트**  — 원인 분석(질문별 detail 근거) + 의사결정
**[5] 오늘 어려웠던 점**
**[6] 내일 목표 / 할 일**
```

## 킥오프 (지금 첫 사이클)
```
python -m scripts.build_index raw
python -m scripts.run_experiments
python -m scripts.log_run
```
→ 그런 다음 업무일지.md에 "Day 8 — 원본 재빌드 + raw 베이스라인" 항목 append.
→ 결과 요약 보고 후 백로그 2번(top_k 스윕)으로 진행.

## 병렬 EDD 실험 루프 (2026-07-06 추가)

검색시점 옵션(top_k, mmr_lambda, fetch_k, auto_filter, rewrite_query)은 raw 인덱스를 다시 만들지 않으므로 병렬 실행 가능하다. 단, `build_index raw`가 필요한 데이터소스/청크크기 실험은 같은 프로젝트 폴더에서 병렬 실행하지 않는다.

1. `eval/parallel_runs/<run>/ledger.json`을 기준으로 run을 만든다.
2. 워커는 `scripts.run_experiment_worker`로 실행하고, `worker_outputs/` 아래에만 CSV/JSON/계약 파일을 쓴다.
3. 메인 코덱스만 `업무일지.md`, `eval/experiment_log.md`, `CLAUDE.md`, `config.py`, `src/`를 수정한다.
4. 집계는 `scripts.aggregate_parallel_eval`로 수행한다.
5. 개선 판단은 원 지표와 함께 EDD score를 본다. EDD는 coverage 20%, hit-all 10%, MRR 15%, groundedness 20%, relevance 20%, abstention 10%, latency 5%의 가중합에서 false abstention rate와 empty answer rate를 페널티로 뺀다.
6. 그래프와 요약은 `summary/summary.md`에 묶고, 가능한 시각화는 전부 남긴다. 현재 기본 그래프는 `edd_score.svg`, `metric_heatmap.svg`, `quality_vs_retrieval.svg`, `latency_vs_edd.svg`.

## 개선 루프 기록 원칙 (사용자 강조사항)

- 많은 실험을 한다. 단, 코호트와 지표가 고정된 상태에서 바꾼 조건이 무엇인지 명확히 남긴다.
- 점수가 올라간 실험만 쓰지 않는다. 점수가 떨어진 실험도 원인, 과정, 결과, 다음 보완 방향까지 반드시 기록한다.
- 매 실험 후 `eval/results` 또는 `eval/parallel_runs`의 질문별 detail JSON을 열어 어느 질문(q1~q10)이 지표를 움직였는지 확인한다.
- 평가지표는 계속 상승하도록 루프를 돈다. 하락하면 "실패"가 아니라 다음 개선의 근거로 삼되, 일지에는 솔직하게 쓴다.
- 업무일지는 보고서체보다 일기체에 가깝게 쓴다. 예상, 빗나감, 판단 변화, 찝찝한 점, 다음 시도를 모두 남긴다.

## 코딩 감리 / 레드팀 게이트

코드 변경이 들어가는 작업은 가능하면 메인 코덱스 단독으로 끝내지 않는다. 메인 코덱스는 구현과 최종 merge owner이고, 감리/레드팀은 직접 최종 파일을 덮어쓰지 않는 proposal/evidence 역할로 붙인다.

### 기본 역할
- **구현 담당(메인 코덱스)**: 실제 코드 수정, 최종 설정 반영, 테스트/스모크 실행, 업무일지 기록.
- **감리(auditor)**: 변경 범위, 회귀 위험, 지표 정의, 데이터/키 커밋 위험, protected path 침범 여부를 점검한다.
- **레드팀(red team)**: 실패 케이스를 일부러 찾는다. 빈 답변, false abstention, coverage 0 착시, judge 오판, 토큰 한도, Chroma/인덱스 충돌, 병렬 파일 충돌을 우선 공격한다.

### 필수 절차
1. 코드 변경 전, 바꿀 파일과 기대 지표를 짧게 적는다.
2. 구현은 작게 한다. worker가 있다면 worker는 `worker_outputs/` 아래 제안만 쓰고, 최종 파일은 메인 코덱스만 수정한다.
3. 구현 후 최소 검증을 실행한다.
   - Python 문법: `python -X utf8 -m py_compile ...`
   - RAG 기본 파라미터 확인: `python -X utf8 -c "from src.rag import RAGPipeline; print(RAGPipeline().params)"`
   - 실험/평가 변경이면 저장된 detail JSON 재계산 또는 작은 fixed cohort smoke.
4. 감리 체크를 통과해야 한다.
   - 지표가 실제로 개선됐는가?
   - 떨어진 지표가 있으면 어느 질문에서 왜 떨어졌는가?
   - 결과 파일/그래프/일지가 남았는가?
5. 레드팀 체크를 통과해야 한다.
   - 빈 답변, 잘못된 abstention, coverage 0 착시, API 토큰 부족, 병렬 충돌이 재현되지 않는가?
   - 실패가 있으면 실패 자체를 기록하고 다음 실험으로 넘긴다.
6. 완료 보고 전, 직접 검증한 결과만 말한다. 사용자가 확인할 수 있는 일을 사용자에게 떠넘기지 않는다.

## 5.5급 LLM 품질 감리 레이어

자동 지표가 높아도 실제 답변이 사람 맥락에 맞지 않을 수 있으므로, 최종 후보나 주요 회귀가 있을 때는 5.5급 강한 LLM 감리를 붙인다. 이 감리는 코드를 직접 바꾸지 않고, 질문/검색근거/답변/detail JSON을 읽고 품질 판정과 원인 proposal만 낸다.

### 투입 시점
- EDD 상위 후보를 최종 기본값으로 고정하기 전.
- EDD는 올랐지만 groundedness/relevance 또는 q별 detail이 찝찝할 때.
- q3 멀티토픽, q5 후속질문, q10 환각방지처럼 자동 채점이 흔들리기 쉬운 케이스.
- 프롬프트, abstention 판정, judge 기준, generation 토큰 정책을 바꾼 뒤.

### 감리 입력
- `eval/questions.json`
- 해당 run의 `summary/summary.md`, `summary/*.svg`
- worker `details.json`
- 문제 케이스의 `question`, `retrieved_orgs`, 검색 chunk 일부, 최종 answer, judge reason

### 감리 산출
- `contextual_quality` 0~5: 답변이 실제 사용자 질문 맥락에 맞는가.
- `evidence_fit` 0~5: 검색 근거와 답변이 실제로 맞물리는가.
- `answer_usefulness` 0~5: 답변이 제출/보고서 관점에서 쓸모 있는가.
- `risk_flags`: 환각, 과잉거절, 근거누락, 질문오독, 평가식오판, 너무 긴 답변, 지연 과다.
- `recommended_action`: 채택 / 조건부 채택 / 재실험 / 프롬프트 수정 / 평가식 수정 / 폐기.

### 운영 원칙
- 5.5 감리 점수는 EDD를 대체하지 않는다. EDD는 정량 비교, 5.5 감리는 문맥 품질 보증이다.
- 둘이 충돌하면 일지에 충돌 자체를 쓴다. 예: "EDD는 96점인데 5.5 감리에서 q3 답변이 근거보다 넓다고 지적".
- 낮은 감리 점수는 실패가 아니라 다음 개선 루프의 입력이다.
- 감리 결과도 `eval/parallel_runs/<run>/audits/`에 JSON/MD로 남기고, `업무일지.md`에는 일기처럼 판단 변화를 적는다.

## 보고서 근거 패킷 원칙

보고서는 단순히 "최종 EDD가 높았다"로 쓰면 안 된다. 과제의 핵심은 어떤 문항을 던졌고, 어떤 답이 왔고, 왜 그 답이 납득 가능/불가능했는지, 그 판단 때문에 다음 실험을 어떻게 바꿨는지 보여주는 것이다.

### 반드시 남길 것
- 평가 문항 전체: `eval/questions.json`의 q1~q10 ID, 질문, 의도, target_orgs.
- 각 주요 실험의 답변 샘플: 성공 답변뿐 아니라 실패/애매한 답변도 포함한다.
- 질문별 판정: coverage, first_hit_rank, abstention, groundedness, relevance, 5.5 감리 점수.
- 납득 안 되는 고득점 사례: 예를 들어 검색 지표는 높지만 답변이 질문보다 넓거나, 근거 없는 사업명을 섞거나, "없음" 판단이 과도한 경우.
- 납득 가능한 저득점/하락 사례: 평가식이 과하게 벌점을 줬거나, 실제로는 정상 부정 답변인데 false abstention으로 잡힌 경우.
- 다음 실험으로 이어진 의사결정: "이 실패 때문에 auto_filter를 켰다", "q5 때문에 rewrite를 유지했다", "fetch_k 40은 노이즈를 늘려 폐기했다"처럼 연결한다.
- 시각화: EDD 막대그래프, metric heatmap, retrieval-quality 산점도, latency-vs-score 그래프.

### 보고서 문장 방향
- "지표가 높으므로 성능이 확실하다"가 아니라, "지표는 좋아졌지만 답변 품질 감리에서 이런 한계가 보였고, 그래서 추가 검증을 붙였다"라고 쓴다.
- 자동 지표와 사람/LLM 감리가 충돌한 사례를 숨기지 않는다. 그 충돌이 실험 기반 개선의 핵심 근거다.
- 최종 결론은 "완벽한 RAG"가 아니라 "평가셋과 반복 실험으로 실패 모드를 줄여간 RAG"로 잡는다.

### 다음 병렬 루프
- 보고서용 근거가 부족하면 새 run을 만든다.
- 각 worker는 서로 다른 실험 축을 돌리되, 모든 worker가 `details.json`에 질문/답변/근거/점수를 남겨야 한다.
- 메인 코덱스는 worker 결과를 합쳐 `report_evidence.md/json` 형태로 요약하고, 업무일지에는 일기처럼 판단 변화를 쓴다.

## 출제자 / 응답자 / 채점자 분리 원칙

평가 품질을 높이려면 질문 생성, 답변 생성, 답변 감리를 한 모델/역할에 섞지 않는다.

### 역할 분리
- **질문 생성자(question generator)**: 가능한 강한 추론 모델을 쓴다. 단순 예시 질문이 아니라, 단일문서/후속질문/비교/불명확 기관명/없는 정보/멀티토픽/함정 질문을 다양하게 만든다.
- **응답자(answering system)**: 반드시 현재 평가 대상 RAG 파이프라인이 답한다. 질문 생성 모델이나 감리 모델이 답변을 대신하면 안 된다.
- **채점자/감리자(quality judge)**: 가능한 강한 모델을 쓴다. RAG 답변이 질문 맥락, 검색 근거, 실사용 관점에 맞는지 다시 본다.

### 운영 규칙
1. 질문 생성자는 정답 힌트를 답변 모델에 직접 주지 않는다. `target_orgs`, `target_biz`는 평가용 메타데이터로만 보관한다.
2. 질문은 다양한 실패 모드를 노린다.
   - 정확한 기관명 질문
   - 부정확한 기관명 질문
   - 후속질문에서 기관명을 생략한 질문
   - 두 기관 비교 질문
   - 문서에 없는 정보 질문
   - 답은 있지만 근거가 여러 청크에 흩어진 질문
3. RAG 응답은 모든 질문/답변/검색근거/chunk/org/judge reason과 함께 저장한다.
4. 채점자는 답변을 고치지 않는다. 점수와 문제 원인만 낸다.
5. 보고서에는 "강한 모델이 만든 다양한 질문 → RAG가 답변 → 강한 모델이 맥락 품질 감리" 구조를 명시한다.
6. 새 질문셋을 만들면 기존 질문셋과 섞기 전에 별도 버전으로 보관한다. 지표 상승/하락 비교 시 평가셋이 바뀌었는지 반드시 기록한다.
